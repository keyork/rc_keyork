# 设计说明与 AI 使用说明

## 一、对问题的理解

### 核心问题

业务系统在调用外部供应商 API 时面临三个现实困境：

1. **供应商不稳定**：外部 API 可能超时、限流（429）、短暂宕机，业务系统无法承担同步等待的代价。
2. **调用方不在线**：通知类场景（广告转化回传、消息推送）本身是"fire-and-forget"，调用方不需要等待结果。
3. **结果需要可审计**：通知是否发出、发出几次、最终状态如何，必须有迹可查，用于对账和排障。

因此这个系统要解决的核心问题是：**将不可靠的同步调用，变成可靠的异步投递**，同时保留完整的状态审计能力。

## 二、整体架构与核心设计

### 架构概览

系统分为两个独立可部署的子系统（通过 `ROLE` 环境变量控制），共享同一个二进制：

- **API 层**：接收提交请求，做安全校验，持久化通知记录，发布到消息队列，立即返回 `202 Accepted`。
- **Worker 层**：从队列消费消息，按域名做熔断判断，执行出站 HTTP 调用，处理重试路由，触发结果回调。

两层通过 RabbitMQ 解耦，通过 PostgreSQL 共享状态。

### 核心设计决策

#### 1. 至少一次投递（At-Least-Once）

Worker 使用 RabbitMQ **manual ack** 模式：只有在 HTTP 调用成功 **且** PostgreSQL 状态更新完成后才 ack 消息。Worker 崩溃时消息自动重投，外部供应商可能收到重复通知——这是有意为之的语义选择（见第四节）。

#### 2. 指数退避重试链

通过 RabbitMQ TTL + Dead Letter Exchange 实现 8 级重试，无需独立的调度器或 cron job。每条消息携带自己的延迟信息，TTL 到期后由 RabbitMQ 自动路由回主队列。延迟序列：30s → 1m → 5m → 30m → 2h → 4h → 8h → 死信。

响应码分类：
- `2xx` → 成功 ack
- `429 / 5xx / 超时 / 连接失败` → 重试
- `4xx (非 429)` → 立即失败，不重试（请求本身有问题）

#### 3. 进程内熔断器

每个目标域名独立维护一个熔断器，存在进程内存（`sync.Map`）中。滑动窗口 5 分钟，≥10 次请求且失败率超过 80% 时熔断，Open 状态持续 60 秒后进入 Half-Open 探测。

熔断期间该域名的消息以 30s 延迟重新入队，**不消耗重试次数**。

#### 4. 接口驱动的依赖注入

`db.Store`、`mq.Publisher`、`mq.Consumer` 均为 Go 接口。Worker、API handler 只依赖接口，从不感知底层实现。`MOCK=true` 时注入内存实现，`MOCK=false` 时注入真实适配器——切换只需改 `cmd/server/main.go` 的依赖注入块。

#### 5. SSRF 防护

`target_url` 在 API 层强制校验：必须 HTTPS、禁止 localhost、禁止 RFC-1918 内网地址（10.x、172.16-31.x、192.168.x）。这道防线只在入口做一次，Worker 不重复校验。

#### 6. 僵死任务回收

后台 goroutine 每 5 分钟扫描 `status=processing` 且 `updated_at < now()-10min` 的记录，重置为 `pending` 并重新入队。覆盖"Worker 写完 DB、ack 之前崩溃"这一无法通过 RabbitMQ 重投恢复的场景。

## 三、系统边界

### 选择解决的问题

| 问题 | 解决方式 |
|------|---------|
| 调用方解耦 | 异步队列 + `202 Accepted` |
| 网络抖动、临时限流 | 指数退避重试 |
| 供应商长期不可用 | 熔断器 + dead-letter queue |
| 消息不丢（Worker 崩溃） | Manual ack + 僵死任务回收 |
| 结果可审计 | PostgreSQL 全程持久化 |
| 结果通知调用方 | Callback URL 异步回调 |
| SSRF 攻击防护 | API 层 URL 校验 |

### 明确不解决的问题

**1. 精确一次投递（Exactly-Once）**

精确一次要求供应商侧配合实现幂等——这超出本系统的控制边界。尝试在本系统内实现精确一次（分布式事务、两阶段提交）会引入巨大的复杂度，且仍然无法消除网络层的重复投递。至少一次 + 文档说明调用方需实现幂等，是合理的抽象边界。

**2. 供应商 API 的认证刷新**

调用方提交的是完整的 HTTP 请求（含 `headers`），本系统不感知 token 过期。如果 token 过期导致 401，会被判定为 4xx 直接失败，不重试。调用方需要在 token 过期前重新提交。这是有意为之的职责边界——本服务是透明的投递通道，不是 API 网关。

**3. 供应商 QPS 限流（v1）**

v1 不做显式的 per-domain 令牌桶。429 响应已被路由进重试链，熔断器提供被动背压。主动限流在供应商合同有明确 QPS 上限、或 429 频率显著影响重试队列深度时，再作为 Phase 2 引入，避免过早优化。

**4. 多租户隔离**

`source_system` 字段用于审计标记，但 v1 不做队列级别的租户隔离。所有业务系统共享同一套队列和 Worker 池。如果需要 SLA 差异化（某个业务优先投递），可在 Phase 4 通过拆分队列 + 加权 Worker 实现。

**5. 历史数据归档**

PostgreSQL 按月分区已为归档预留了结构，但 v1 不做 S3 卸载。触发条件：单表数据量超过 5000 万行，或审计查询 P99 超过 2 秒。

## 四、可靠性与失败处理

### 投递语义：至少一次

选择至少一次而非精确一次的理由：

1. **RabbitMQ manual ack 天然提供至少一次语义**，不需要额外机制。
2. **精确一次的代价不对等**：需要幂等 key + 分布式锁 + 两阶段确认，引入的复杂度远超收益，且仍依赖供应商侧配合。
3. **重复投递的实际影响可控**：在 Worker 崩溃这一小概率场景下，重复投递才会发生；正常流程下 ack 在成功后立即发出，不会重复。

### 外部系统失败策略

| 场景 | 策略 |
|------|------|
| 供应商单次超时/5xx | 进入重试链，最长持续 ~14.6 小时 |
| 供应商持续不可用（>10 次失败，失败率>80%） | 熔断器 Open，消息暂存在 30s 重试队列，不消耗重试次数 |
| 供应商长期不可用（重试耗尽） | 进入死信队列，等待人工介入或调用方 manual retry |
| 供应商返回 4xx（请求错误） | 立即标记 failed，不重试；避免用错误请求反复打扰供应商 |
| RabbitMQ 节点故障 | Quorum queue 要求多数节点确认，单节点宕机不丢消息；Worker 重连后继续消费 |
| PostgreSQL 故障 | Worker 无法更新状态时不 ack，消息重投；API 层 Create 失败时返回 500，调用方可重试提交 |
| Worker 进程崩溃（DB写后/ack前） | 僵死任务回收（5min扫描）重置并重投 |

## 五、取舍说明

### 选择 RabbitMQ 而非 Kafka

引入消息队列的出发点有三个，都是在技术方案讨论阶段确定的：

1. **审计与分析需求**：每条通知的投递状态必须持久化可查，消息队列提供了 API 层与 Worker 之间的异步缓冲，使两侧可以独立扩缩容，同时通过 PostgreSQL 保留完整状态流转记录。
2. **业务方需要投递结果**：调用方提交通知后不能同步等待外部 API 返回，需要通过 callback 或主动 poll 获取最终状态。消息队列是实现这一解耦的必要基础设施。
3. **缓冲队列保障可靠性**：外部供应商 API 不稳定，需要一个可靠的缓冲层在失败时持有消息，而不是让调用方自己管理重试。这个职责天然属于消息队列。

在此基础上选择 RabbitMQ 而非 Kafka 的理由：这是任务分发模型，而非事件流模型。每条消息由一个 Worker 处理，不需要多消费组重放。RabbitMQ 的 TTL + DLX 天然支持延迟重试链；Kafka 实现等价语义需要自建 retry topic 链，且 Consumer 数量受 partition 数限制，扩缩容不如 RabbitMQ 灵活。

### 选择进程内熔断而非 Redis

熔断判断在每次 HTTP 调用的热路径上同步执行。Redis 网络查询在高并发下增加 1–5 ms 额外延迟，在 Worker 并发 100 的场景下是不可接受的开销。多实例熔断状态短暂不一致是可接受的代价——单个实例的网络抖动不会通过全局熔断器误伤其他实例的正常投递。

### HTTP 层使用标准库而非 Gin

HTTP 层直接使用 Go 标准库 `net/http` + `ServeMux`，不引入第三方框架。当前只有 5 个端点，路由规则简单，Gin 的路由分组、中间件链等能力在此规模下没有实质收益，引入它只会带来项目唯一的外部依赖，增加版本管理负担。

如果端点数量持续增长、需要统一的请求日志/鉴权/限流中间件，迁移到 Gin 是合理的演进——Gin 的 `gin.Context` 与标准库接口差异不大，迁移成本可控。这也是在与 AI 讨论技术方案时明确的判断：当前规模不值得换，未来规模到了自然会换。

### 接口 + Mock 优先，而非真实基础设施

v1 阶段真实适配器（PostgreSQL、RabbitMQ）以接口占位符形式存在，所有测试和本地运行使用内存实现。这个决策大幅降低了开发环境搭建成本，也使单元测试完全隔离、可确定性运行，不依赖外部进程。接口设计保证切换到真实适配器时其他代码零改动。

## 六、未来演进路径

系统在以下触发条件下需要演进：

| 阶段 | 触发条件 | 演进方向 |
|------|---------|---------|
| Phase 1 | 表行数 > 5000 万，审计查询变慢 | 月度分区 + 90 天数据归档至 S3（Parquet），Athena 按需查询 |
| Phase 2 | 某供应商 429 频繁，或合同有明确 QPS 上限 | per-domain 令牌桶；多实例时引入 Redis 做集中限流 |
| Phase 3 | 多地域部署，有区域延迟 SLA 要求 | 各区域独立 RabbitMQ 集群，PG 单主 + 异步只读副本 |
| Phase 4 | 不同业务场景对投递时效有差异化 SLA | 主队列拆分 high/normal/low，Worker 加权消费 |
| Phase 5 | 端点持续增多，中间件需要统一管理 | 迁移到 Gin，使用路由分组、中间件链和请求绑定 |

## 七、AI 使用说明

### AI 在哪些关键环节提供了帮助

**1. 系统骨架快速成型**

基于技术设计文档（`docs/technical_proposal.md`），AI 生成了完整的 Go 项目结构：接口定义、内存 Mock 实现、API handler、Worker 池、熔断器、配置加载、优雅退出。这将原本需要数天的脚手架工作压缩到数小时。

**2. 发现并修复了若干非显而易见的 Bug**

- **`pathSegment` 索引错误**：`GET /api/v1/notifications/{id}` 路径被 `strings.Trim` 后有 4 段（索引 0–3），ID 在索引 3，而原实现取的是索引 4，导致 Get 永远返回空 ID 进而报 404。AI 在补充 `TestGetExisting` 测试时发现并修正。
- **测试中的 TLS 握手失败**：Worker 测试最初使用 `httptest.NewTLSServer`，但 Worker 的 `http.Client` 不信任自签证书，导致所有投递在测试中失败。AI 将其改为 `httptest.NewServer`（明文 HTTP），SSRF/HTTPS 校验是 API 层的职责，Worker 不需要重复执行。
- **Mock MQ 的无限重试循环**：`PublishRetry` 最初会立即将消息重新放回主队列，导致 5xx 测试中 Worker 进入无限循环。AI 将其改为"记录但不重投"，测试需要重投时显式调用 `Requeue()`，使测试行为确定。
- **4xx 被错误判定为成功**：`deliver()` 最初以 `err == nil` 判断成功，但 4xx 响应同样返回 `(statusCode, nil)`。AI 将判断改为显式检查状态码：`2xx` 成功，非 429 的 4xx 立即失败，其余重试。

**3. 结构化日志体系**

将全部 `fmt.Printf` / `log.Printf` 替换为 `log/slog` 分级结构化日志，统一 `component` 字段，制定 DEBUG/INFO/WARN/ERROR 使用规范，并编写 `internal/logger/logger.go` 封装初始化逻辑。

**4. 文档生成**

README.md（英文）、README_CN.md（中文）、`docs/runbook.md`（运行手册含预期日志示例）均由 AI 生成初稿并根据反馈迭代。

### AI 给出过但未采纳的建议

**1. 搭建真实基础设施（docker-compose + 数据库迁移）**

AI 开始创建包含真实 RabbitMQ 和 PostgreSQL 的 `docker-compose.yml` 及数据库迁移脚本。这被明确拒绝：v1 阶段只需要内存 Mock，真实基础设施会增加本地开发门槛，也会让测试依赖外部进程。接口设计已经为未来切换真实适配器留好了扩展点，不需要现在就实现。

**2. 隐式的通用性扩展**

在部分代码实现中，AI 倾向于加入过度的抽象（如通用的重试策略接口、可插拔的熔断后端），这些都被拒绝或简化。v1 只需要能跑通的最小实现，过早抽象带来的维护成本高于收益。

---

### 关键决策是自己做出的

**1. 引入消息队列及选型 RabbitMQ**

在技术方案讨论阶段确定引入消息队列，驱动因素是三个明确的业务需求：审计分析（每条通知的状态必须持久化可查）、业务方需要异步获取投递结果（不能让调用方同步等待）、以及对不稳定外部 API 的缓冲保障。在此基础上，通过与 AI 的方案对比讨论，选定 RabbitMQ 而非 Kafka——核心理由是 TTL + DLX 原生支持延迟重试链，且任务分发模型不需要 Kafka 的多消费组重放语义。

**2. Mock 优先、不引入真实中间件**

这是整个开发模式的核心约束。放弃 docker-compose 不是技术限制，而是主动选择：让系统在任何环境下 `make run` 就能跑起来，让测试不依赖进程，让接口设计驱动实现而非反过来。

**3. 系统边界划定**

精确一次、Token 刷新、多租户隔离、供应商速率限制——这些都是有意排除在 v1 之外的。每一条都有明确的判断依据（见第三节），而不是遗忘或偷懒。判断标准是：这个问题是否在本系统的控制边界内？引入的复杂度是否与当前规模匹配？

**4. 重试计数在熔断期间不递增**

熔断期间该域名的消息以最短延迟暂存，不消耗 `retry_count`。这个细节 AI 在实现中没有自动处理，是在设计评审时补充的：熔断是对"供应商整体不可用"的响应，不应该计入单条通知的投递失败记录，否则会导致大量通知在供应商恢复后因"重试次数已满"而进入死信，产生数据丢失。

**5. Callback 使用独立的简单重试，而非复用主重试链**

Callback 失败最多重试 3 次（延迟 0s/1s/5s/30s），之后放弃，调用方可 poll API 兜底。刻意不将 callback 路由进 RabbitMQ 重试链：一方面避免递归复杂度（通知投递失败 → callback 投递失败 → callback 的 callback？），另一方面 callback 的重要性低于主通知投递，不值得占用同等的重试配额。
